# Нейра — Саморазвивающийся символьный ИИ-агент

## О проекте

**Нейра** — экспериментальный ИИ-агент, который строится не на статистическом угадывании, а на явных правилах понимания, проверяемых гипотезах и управляемом накоплении знаний.

Проект создаётся как **исследовательская система**, а не как игровой NPC. Unreal Engine здесь используется как удобный фреймворк, среда интеграции и визуальный слой.

- **Язык ядра:** C++
- **Среда:** Unreal Engine
- **Визуальный слой:** Blueprints
- **Главный язык анализа:** русский
- **Второй язык:** английский

## Философия

- Агент **не угадывает**, а разбирает вход по правилам.
- Самообучение идёт через **гипотезы, подтверждения и пересмотр**, а не через слепую фиксацию выводов.
- Ошибки не должны цементироваться: знания могут не только усиливаться, но и **деградировать**.
- Омонимия и неоднозначность считаются **нормой языка**, а не ошибкой системы.
- Внешние LLM допустимы как вспомогательный инструмент обучения и проверки, но не как runtime-мозг.

## Scope и границы проекта

### Что входит в проект

- интерпретация естественного языка;
- морфологический и далее синтаксический разбор;
- хранение гипотез, знаний и контекстных интерпретаций;
- управляемая валидация;
- построение ответа на основе внутренних правил;
- интеграция логики с Unreal / Blueprint.

### Что **не** является целью на текущем этапе

- AGI / полноценный общий интеллект;
- автономная самомодификация кода без строгой верификации;
- свободное самообучение без внешнего контроля;
- продакшн-замена LLM;
- немедленный запуск сложного семантического графа;
- полный морфемный разбор русского языка в v0.1.

## Целевая архитектура

Это **долгосрочная карта**, а не порядок немедленной реализации.

```text
Слой 0: Лингвистическое ядро        — словари, таблицы, правила, признаки
Слой 1: Морфологический анализатор  — форма слова → лемма → часть речи
Слой 2: Синтаксический парсер       — структура предложения и связи
Слой 3: Семантический граф          — понятия и отношения
Слой 4: Механизм рассуждений        — вывод, выбор действий, память
Слой 5: Генератор ответа            — формирование ответа по правилам
```

### Оценка веса модели

| Компонент | Размер |
|-----------|--------|
| Слой 0: словари + правила | ~100–150 МБ |
| Слои 1–2: код + таблицы правил | ~10–20 МБ |
| Слой 3: семантический граф (старт) | ~10–50 МБ |
| **Итого на старте** | **~200–300 МБ** |
| Семантический граф (рост) | до гигабайтов |

## План сборки

### v0.1 — минимальный интерпретатор фраз

Без словарей, на паттернах и базовых правилах.

- тип фразы: вопрос / команда / утверждение / просьба;
- намерение: узнать / выполнить / проверить возможность / уточнить;
- выделение объекта действия;
- фиксация причины вывода и уровня уверенности;
- накопление гипотез без записи в жёсткие знания.

### v0.2 — лёгкая морфология

- словарь-ядро: 5 000–10 000 частотных слов;
- леммы и части речи;
- определение части речи: сначала по словарю, потом по признакам;
- базовая обработка омонимии через контекст;
- первые устойчивые правила перехода гипотез в знания.

### v0.3 — синтаксические связи

- кто → действие → объект → адресат;
- вложенные действия;
- различение прямого действия и проверки возможности;
- контекстный выбор интерпретации слова.

#### Критерии готовности v0.3 (Definition of Done)

v0.3 считается готовой только если одновременно выполнены все условия:

1. **Синтаксический разбор воспроизводим**  
   Для контрольного набора фраз (включая омонимы) парсер стабильно строит одинаковые связи `subject/action/object/recipient`.
2. **Выбор интерпретации объясним**  
   Для каждого AmbiguousToken сохраняется trace: какие якоря/маркеры сработали, какой confidence и почему выбран итоговый вариант.
3. **Память деградирует управляемо**  
   При смене темы и при давлении памяти (`Medium/High/Critical`) нет потери целостности: остаётся либо HOT-контекст, либо WARM-summary, либо COLD+anchor.
4. **Fallback остаётся рабочим**  
   При ошибках синтаксиса система не падает в `EMPTY_INPUT`, а возвращает частичный разбор + корректный fail reason.
5. **Есть регрессионный gate**  
   Любое изменение порогов (`topic_change_threshold`, confidence thresholds) проходит сравнение на фиксированном наборе RU/EN-кейсов.

### v0.4 — управляемое самообучение

- гипотезы, стабильные гипотезы, проверенные знания;
- переходы вверх и вниз;
- внешняя валидация;
- защита от закрепления ложных выводов.

### v0.5 — семантический граф

- только после стабилизации нижних слоёв;
- связи между понятиями;
- уплотнение, дедупликация, проверка чистоты данных.

### Далёкий этап — понимание C++ и самомодификация

Это отдельный крупный подпроект: лексер → парсер → AST → генерация кода → runtime compilation → hot reload → верификация. Не является продолжением языкового ядра.

## Рабочий pipeline агента

Базовый цикл работы системы:

```text
Ввод
 → Нормализация строки
 → Токенизация
 → Анализ слов / гипотез
 → Выбор интерпретаций по контексту
 → Определение типа фразы
 → Определение намерения
 → Выбор действия
 → Выполнение действия
 → Формирование ответа
 → Обновление памяти и статусов гипотез
```

---

## Пути отказа в pipeline

Каждый шаг pipeline может не дать результата. Система не должна падать — она должна деградировать управляемо.

### Сбои на этапах pipeline

| Этап | Сбой | Реакция |
|---|---|---|
| Нормализация / Токенизация | Пустой ввод или только пунктуация | Вернуть `EMPTY_INPUT`, ответить «Не услышала ничего» |
| Морфология | Ноль интерпретаций для токена | Пометить как `UnknownToken`, продолжить с остальными |
| Морфология | Все токены `UnknownToken` | Эскалация (см. ниже) |
| Интерпретация | Все confidence < 0.3 | Сохранить лучшую как `AmbiguousToken`, низкий opener в ответе |
| Определение намерения | Intent не распознан | Установить `UNKNOWN_INTENT`, эскалация |
| Выбор действия | ActionID не зарегистрирован | `FailReason::NotSupported` |
| Выполнение действия | Объект не найден | `FailReason::NotFound` |

Правило: если хотя бы одна часть фразы разобрана — продолжаем. Полная остановка только при `EMPTY_INPUT`.

---

### Эскалация при неизвестном вводе

Когда агент не может справиться своими средствами, он эскалирует по цепочке:

```text
Сбой (UNKNOWN_INTENT или все токены UnknownToken)
    │
    ├─ 1. Проверить HOT/WARM MEMORY — видели похожее?
    │       └─ Нашли → использовать прошлый результат, confidence снизить
    │
    ├─ 2. Проверить ANCHOR STORAGE — есть тема?
    │       └─ Нашли → достать якорь, уточнить у пользователя
    │
    ├─ 3. Запросить внешнюю нейросеть (LLM)
    │       └─ Ответ получен → сохранить как Hypothesis
    │                          с меткой source = ExternalLLM
    │                          weight = 0.7 (не может стать VerifiedKnowledge)
    │
    └─ 4. Запросить разработчика (флаг needs_developer_review)
            └─ Разработчик подтвердил → может перейти в VerifiedKnowledge
```

---

### Метка ExternalLLM

Ответ от внешней нейросети всегда помечается явно — в ответе агента и в памяти.

```text
В ответе пользователю:
  «[Нейросеть] Слово "X" означает...»
  — пользователь видит, что это не знание агента

В памяти:
  FWordInterpretation.SourceType = ExternalLLM
  FWordInterpretation.State      = Hypothesis   ← не выше
  FWordInterpretation.SourceWeight = 0.7
```

Ответ внешней нейросети **не может** автоматически стать `VerifiedKnowledge`. Это заблокировано архитектурно: переход в `VerifiedKnowledge` через внешний LLM запрещён — только через подтверждение разработчика или словарь.

---

### Флаг needs_developer_review

Если агент сохранил гипотезу с меткой `ExternalLLM` или столкнулся с неизвестным паттерном, он ставит флаг `needs_developer_review`.

```text
FPendingReview
├── InputPhrase    : FString         — исходная фраза
├── FailStage      : EpipelineStage  — на каком этапе споткнулись
├── ExternalAnswer : FString         — что ответила внешняя нейросеть (если было)
├── SourceType     : ESourceType
└── CreatedAt      : FDateTime
```

Разработчик просматривает очередь `PendingReviews` и подтверждает или отклоняет. Подтверждение → `VerifiedKnowledge`. Отклонение → `Hypothesis` остаётся, флаг снимается.

## Интерфейс морфология → синтаксис

### Принятое решение

Морфологический анализатор обрабатывает **все токены сразу** и передаёт синтаксическому парсеру **полный список** со всеми возможными интерпретациями.

Синтаксис работает с полной картиной предложения — это позволяет разрешать омонимию с учётом контекста всего предложения, а не только предшествующих слов.

```text
Токенизация
 → Морфология (все токены сразу)
    → [токен1: [интерп1, интерп2], токен2: [...], ...]
       → Синтаксис (полная картина)
          → выбор активной интерпретации по контексту
```

### Почему не пошагово (токен за токеном)

Русский язык имеет свободный порядок слов. Интерпретация слова в начале предложения часто определяется словами в конце:

```text
«Печь жарко топили» → «печь» = существительное (топили что? = печь)
«Печь пироги умела»  → «печь» = глагол (умела что? = печь)
```

При пошаговой обработке синтаксис получает `«печь»` раньше, чем видит `«топили»` или `«пироги»` — разрешить омонимию невозможно.

### Триггер для выбора подхода

| Условие | Подход |
|---|---|
| Язык с фиксированным порядком слов | Пошаговая обработка допустима |
| Язык со свободным порядком слов (русский) | Полная картина обязательна |
| Омонимия — ключевая задача системы | Полная картина обязательна |

Для Нейры: омонимия — центральная задача, русский — свободный порядок слов. Полная картина обязательна.

## Механизм понимания фраз

Базовая схема интерпретации:

```text
Фраза → Тип фразы → Намерение → Целевое действие → Объект действия
```

### Примеры

| Фраза | Тип | Намерение | Действие | Объект |
|---|---|---|---|---|
| Дай определение слова мама | команда | получить знание | найти определение | слово «мама» |
| Ты можешь нарисовать квадрат? | вопрос | проверить возможность | ответить о возможности | нарисовать квадрат |
| Кайл боится лабораторий | утверждение | сохранить факт | запомнить | Кайл / страх / лаборатории |
| Найди значение слова дом | команда | получить знание | поиск значения | слово «дом» |

## Генератор ответа

Генератор ответа — последний шаг pipeline. Получает на вход структурированный результат обработки фразы и строит текстовый ответ агента.

```text
Intent + ActionResult + Confidence + DialogContext
  → ResponseGenerator
    → строка ответа на естественном языке
```

### Архитектура: три слоя

```text
┌─────────────────────────────────────────────────────────────┐
│                   FResponseGenerator                         │
│                                                              │
│  ┌───────────────┐   ┌───────────────┐   ┌───────────────┐ │
│  │  Compositor   │→  │   Selector    │→  │    Learner    │ │
│  │ собирает      │   │ выбирает      │   │ обновляет     │ │
│  │ структуру     │   │ вариант по    │   │ веса по       │ │
│  │ ответа        │   │ весам         │   │ сигналу       │ │
│  └───────────────┘   └───────────────┘   └───────────────┘ │
└─────────────────────────────────────────────────────────────┘
```

---

### Слой 1: Compositor

Разбивает ответ на независимые компоненты. Каждый компонент генерируется по правилам из входных данных.

```text
Структура ответа: [opener] + [content] + [closer]
```

#### opener — вводная часть (опциональная)

Определяется уровнем уверенности агента:

| Confidence | Opener |
|---|---|
| > 0.8 | *(пусто)* |
| 0.5–0.8 | «Кажется, » |
| 0.3–0.5 | «Не уверена, но » |
| < 0.3 | «Я не совсем понимаю, но » |

#### content — основное содержание

Генерируется из `Intent + ActionResult`. Ключевые сущности подставляются в нужном падеже — для этого content-слой обращается к морфологическому модулю.

| Intent | Шаблон content |
|---|---|
| `GET_DEFINITION` | «{entity[им.п.]} — {definition}» |
| `STORE_FACT` | «Запомнила: {subject[им.п.]} {predicate} {object[вин.п.]}» |
| `CHECK_ABILITY` | «{ability[вин.п.]} я {могу/не могу}» |
| `SEARCH_MEANING` | «{entity[им.п.]} означает {meaning}» |
| `UNKNOWN_INTENT` | «Не поняла про {best_guess_entity[вин.п.]}» |

#### closer — завершающая часть (опциональная)

Определяется диалоговым контекстом:

| Условие | Closer |
|---|---|
| агент не уверен в интерпретации | «, правильно понимаю?» |
| есть связанная информация | «Хочешь узнать больше?» |
| ответ полный и уверенный | *(пусто)* |

---

### Слой 2: Selector

Для каждого компонента (`opener`, `content`, `closer`) хранится несколько вариантов формулировки с весами. Selector выбирает вариант взвешенным случайным образом.

```text
content для GET_DEFINITION:
  ├── «{entity} — это {definition}»                weight: 1.0
  ├── «Значение слова {entity}: {definition}»       weight: 1.0
  └── «{entity} означает {definition}»              weight: 1.0

→ по мере накопления диалогов веса расходятся
```

Selector также учитывает **стилевой профиль пользователя** (из слоя гипотез):

| Профиль пользователя | Эффект |
|---|---|
| `prefers_short_answers: true` | убрать opener и closer |
| `asks_follow_up: часто` | усилить вероятность closer с вопросом |
| `formality: informal` | предпочитать разговорные формулировки |

---

### Слой 3: Learner

Обновляет веса после каждого ответа на основе **имплицитного сигнала** — следующей фразы пользователя. Явного «оцени ответ» не требуется.

```text
Сигналы и их интерпретация:

  следующая фраза                         → сигнал
  ─────────────────────────────────────────────────
  «спасибо», «понял», «ок»               → +reward
  продолжение темы (семантически близко) → +reward
  «что?», «повтори», «не понял»          → -reward
  переспрос с уточнением той же сущности → -reward
  немедленная смена темы                 → нейтральный
  долгое молчание                        → слабый -reward
```

Обновление веса выбранного варианта:

```text
new_weight = old_weight + learning_rate * reward
```

`learning_rate` — небольшое значение (0.05–0.1), чтобы обучение было плавным и не переобучалось на единичных сигналах.

---

### Case-Based Reasoning (CBR) — резервный путь

Если Compositor не может построить content для нового типа намерения (не хватает правил), система обращается к базе прошлых диалогов.

```text
Алгоритм CBR:
  1. Найти в базе случай с наиболее похожим Intent + entities
     (сравнение по набору сущностей, без LLM)
  2. Взять response из найденного случая
  3. Заменить сущности на актуальные (с правильными падежами)
  4. Если совпадение < порога → вернуть UNKNOWN_INTENT ответ

Обучение CBR:
  каждый успешный диалоговый обмен (+reward) сохраняется в базу случаев
  база случаев → COLD MEMORY на диске
```

### Интерфейс генератора

```text
Вход:
  FResponseInput
  ├── intent            — распознанное намерение
  ├── action_result     — результат выполнения действия
  ├── confidence        — уверенность интерпретации [0..1]
  ├── key_entities      — ключевые сущности с падежами
  └── dialog_context    — последние N фраз из context_window

Выход:
  FResponseOutput
  ├── text              — итоговая строка ответа
  ├── components_used   — какие компоненты вошли (для Learner)
  └── variants_selected — индексы выбранных вариантов (для Learner)
```

## Типы памяти

Проекту нужна не одна общая «память», а несколько разных слоёв хранения.

### 1. Оперативная память

Хранит текущее состояние обработки:
- входную фразу;
- токены;
- промежуточные гипотезы;
- активное намерение;
- выбранное действие.

Структура слотов (вариант B — отдельные слоты на каждый этап pipeline):

```text
FAgentWorkingMemory
├── morphology_result    — выход морфологии
├── syntax_result        — выход синтаксиса
├── semantic_result      — выход семантики
└── context_window       — TArray последних фраз (динамический размер)
```

Контекстное окно хранит несколько последних фраз для сохранения диалогового контекста. Размер окна определяется динамически через `FMemoryMonitor`.

#### Многоуровневая модель памяти

```text
┌─────────────────────────────────────────────────────┐
│                  FMemoryMonitor                      │
│  (следит за RAM, управляет переходами между слоями) │
└──────────┬──────────────────┬───────────────────────┘
           │                  │
           ▼                  ▼
┌──────────────────┐   ┌──────────────────────────────┐
│   HOT MEMORY     │   │        TRIGGER ENGINE         │
│  (RAM, полный    │   │  - смена темы (семант. сдвиг) │
│   контекст)      │   │  - явный маркер ("кстати...")│
│                  │   │  - долгая пауза              │
│  N последних     │   └──────────┬───────────────────┘
│  FAgentWorking   │              │ тема сменилась
│  Memory          │              ▼
└────────┬─────────┘   ┌──────────────────────────────┐
         │ RAM кончается│        ANCHOR CREATOR         │
         ▼             │  Сжать контекст → Anchor      │
┌──────────────────┐   │  (ключевые сущности + вектор) │
│   WARM MEMORY    │   └──────────┬───────────────────┘
│  (сжатые        │              │
│   саммари,      │              ▼
│   ещё в RAM)    │   ┌──────────────────────────────┐
└────────┬─────────┘   │      ANCHOR STORAGE (RAG)    │
         │ совсем мало │  - индекс якорей по темам    │
         ▼             │  - семантический поиск       │
┌──────────────────┐   │  - "вернись к теме X →      │
│   COLD MEMORY    │   │    достань якорь X"          │
│  (диск/БД,      │   └──────────────────────────────┘
│   полные логи)  │
└──────────────────┘
```

#### FMemoryMonitor — политика давления

```text
EMemoryPressure:
  Low      → хранить полный FAgentWorkingMemory за каждую фразу
  Medium   → сжимать: оставлять только итог + intent
  High     → выгружать в COLD, оставлять якорь
  Critical → только текущий цикл, всё остальное на диск
```

#### FTopicChangeDetector — триггер смены темы

Смена темы — основной триггер очистки HOT MEMORY. При смене темы агенту нет смысла держать весь предыдущий контекст в RAM.

##### Векторное представление фразы

Используется TF-IDF по частотному словарю (подключается вместе с OpenCorpora в v0.2).

```text
TF(лемма, фраза)  = count(лемма в фразе) / count(все леммы в фразе)
IDF(лемма)        = log(N / df(лемма))
                    N     — число фраз в context_window
                    df    — число фраз, содержащих лемму

вектор фразы = { TF-IDF(лемма) для каждой леммы из частотного словаря }
```

Размерность вектора = размер частотного словаря. В памяти хранится как `TArray<float>`, не `FVector` (3D здесь не подходит).

IDF обновляется при каждом добавлении фразы в `context_window` — накапливается по ходу диалога.

##### Алгоритм детекции

```text
Новая фраза добавлена в context_window
    │
    ├─ Вычислить TF-IDF вектор новой фразы
    ├─ Вычислить средний TF-IDF вектор context_window (без новой фразы)
    ├─ Косинусное расстояние > порога?
    │       НЕТ → сбросить pending_topic_change, продолжить
    │       ДА  → установить pending_topic_change = true
    │
    └─ pending_topic_change уже был true на предыдущей фразе?
            НЕТ → ждать следующей фразы
            ДА  → подтвердить смену темы → создать Anchor → очистить HOT
```

Смена темы подтверждается только если **две фразы подряд** дали расстояние выше порога. Это защищает от ложных срабатываний на разовых отклонениях («да», «понял», «хорошо»).

##### Стартовый порог

Стартовое значение порога: `0.6`. Калибруется на реальных диалогах начиная с v0.3.

Дополнительные триггеры (не требуют вектора):
- явный маркер в речи («кстати», «вопрос не по теме», «а ещё»);
- долгая пауза между фразами.

#### FAnchorSystem — мини-RAG

При смене темы контекст не уничтожается, а сворачивается в якорь и сохраняется в индекс. Якорь позволяет вернуться к теме позже.

```cpp
struct FMemoryAnchor {
    FString            TopicSummary;    // "обсуждали настройки графики"
    TArray<FString>    KeyEntities;     // ["fps", "качество", "настройки"]
    TArray<float>      SemanticVector;  // TF-IDF вектор для поиска по якорям
    FDateTime          Timestamp;
    FString            ColdStoragePath; // путь к полному логу темы
};
```

#### Алгоритм поиска по якорям (retrieval)

`FAnchorSystem` ищет не «идеальное совпадение», а лучший кандидат по нескольким сигналам:

```text
Вход: новая фраза пользователя
  │
  ├─ 1) Нормализовать и лемматизировать
  ├─ 2) Построить query-vector (TF-IDF)
  ├─ 3) Отобрать top-K кандидатов по косинусной близости
  ├─ 4) Пересчитать score кандидатов:
  │       score = 0.70 * semantic_similarity
  │             + 0.20 * entity_overlap
  │             + 0.10 * recency_decay
  ├─ 5) Проверить пороги:
  │       score >= 0.72  → AUTO_RESTORE_TOPIC
  │       0.55..0.72     → ASK_USER_CONFIRMATION
  │       < 0.55         → NO_MATCH
  └─ 6) При восстановлении темы:
          загрузить связанный COLD-лог и поднять краткий WARM-summary в HOT
```

Где:
- `semantic_similarity` — косинусная близость между query-vector и `SemanticVector` якоря;
- `entity_overlap` — доля совпавших ключевых сущностей (`KeyEntities`);
- `recency_decay` — коэффициент давности (свежие темы получают небольшой бонус).

Правило безопасности: если совпало несколько якорей с разницей `score < 0.05`, автоподъём запрещён — только уточняющий вопрос пользователю.

#### Цикл работы памяти

```text
Новая фраза
    │
    ├─► FTopicChangeDetector
    │       │ смена темы
    │       └─► FAnchorCreator → сохранить якорь → очистить HOT
    │
    ├─► FMemoryMonitor
    │       │ давление HIGH/Critical
    │       └─► сжать WARM → COLD, оставить якорь
    │
    └─► добавить в HOT MEMORY (context_window)
```

#### Порядок реализации

| Приоритет | Компонент | Этап |
|-----------|-----------|------|
| 1 | `FMemoryMonitor` — проверка RAM + политика давления | v0.3 |
| 2 | `FAgentWorkingMemory` — слоты + context_window | v0.3 |
| 3 | `FTopicChangeDetector` — детектор смены темы | v0.3 |
| 4 | `FAnchorSystem` — хранение и поиск якорей | v0.4 |

## Модель данных слова

### Принцип

Слово не равно одной трактовке. Одна форма может иметь несколько валидных интерпретаций. Это не исключение, а системное свойство русского языка.

### Двухуровневая схема

```text
FWordEntry (запись слова)
├── Text: FString                          — исходная форма
├── LemmaCandidates: TArray<FString>       — возможные леммы
├── Interpretations: TArray<FWordInterpretation>
└── CreatedAt / UpdatedAt                  — метаданные

FWordInterpretation (одна интерпретация)
├── WordType: EWordType                    — часть речи
├── State: EKnowledgeState                 — статус знания
├── Confidence: float                      — уверенность 0.0–1.0
├── Reason: FString                        — почему так решил
├── SourceWeight: float                    — вес источника
├── Confirmations: int32                   — число подтверждений
├── Rejections: int32                      — число опровержений
├── ContextMarkers: TArray<FString>        — когда эта интерпретация активна
└── History: TArray<FStateChange>          — история изменений статуса
```

### Пример

```text
Слово: «печь»
  Интерпретация 1:
    тип: существительное
    статус: VerifiedKnowledge
    уверенность: 0.95
    источник: словарь (вес 0.9)
    подтверждений: 34 / опровержений: 0
    контекст: после предлога, после прилагательного

  Интерпретация 2:
    тип: глагол
    статус: VerifiedKnowledge
    уверенность: 0.92
    источник: словарь + контекст (вес 0.9)
    подтверждений: 51 / опровержений: 0
    контекст: после модального глагола, перед сущ. в вин. падеже
```

Выбор активной интерпретации — задача синтаксического слоя (v0.3), описана в разделе «Механизм выбора интерпретации».

## Механизм выбора интерпретации

Синтаксис получает полный список токенов, каждый с несколькими интерпретациями. Задача — выбрать одну активную интерпретацию для каждого токена.

### Алгоритм: два прохода с якорями

#### Проход 1 — найти якоря

Якорь — токен, интерпретация которого однозначна:
- единственная интерпретация, или
- `VerifiedKnowledge` + `Confidence > 0.9`.

```text
«Ты можешь печь пироги»
  «Ты»     → местоимение (один вариант) ← якорь
  «можешь» → глагол, модальный           ← якорь
  «печь»   → сущ. / глагол              ← спорный
  «пироги» → существительное            ← якорь
```

#### Проход 2 — якоря разрешают соседей через маркеры

Каждая интерпретация хранит `ContextMarkers` — предикаты о соседних токенах:

```text
ContextMarker = {
    Direction: Left | Right
    Distance:  1 | 2 | Any
    Condition: WordType | MorphFeature
}
```

Примеры для слова «печь»:

```text
Интерпретация: существительное
  Маркеры: [{Left, 1, Preposition}, {Left, 1, Adjective}]

Интерпретация: глагол
  Маркеры: [{Left, 1, Modal}, {Right, 1, Noun_Accusative}]
```

Якорь `«пироги»` (Noun) справа → голосует за глагол.
Якорь `«можешь»` (Modal) слева → тоже голосует за глагол.
Результат: `«печь»` = глагол.

#### Если маркеры не дали однозначного ответа

```text
Маркеры не сработали
    │
    ├── одна интерпретация с явно выше Confidence → берём её
    │
    ├── ничья → оставляем обе, маркируем токен как AmbiguousToken
    │           (это нормально, не ошибка)
    │
    └── нет маркеров вообще → только Confidence, логируем причину
```

### Откуда берутся маркеры

Маркеры хранятся в словаре как часть интерпретации. Источники:

| Источник | Этап |
|---|---|
| Словарь OpenCorpora | v0.2 |
| Вручную для ядра омонимов (регрессионные кейсы) | v0.2 |
| Самообучение: подтверждённый контекст → новый маркер | v0.4 |

### Итог

| Вопрос | Ответ |
|---|---|
| Что такое маркер? | Предикат о соседнем токене |
| Кто их применяет? | Синтаксис, второй проход |
| Проблема «курица и яйцо»? | Снимается якорями — сначала однозначные токены |
| Что если не совпало? | Confidence → AmbiguousToken → лог |

## Состояния знаний

```text
Hypothesis          — начальная догадка, может быть ошибочной
StableHypothesis    — подтверждена несколько раз, мало опровержений
VerifiedKnowledge   — проверена сильным источником
```

### Переходы вверх

```text
Hypothesis → StableHypothesis:
  - Confidence > 0.7
  - Confirmations >= 3
  - Rejections < Confirmations * 0.2

StableHypothesis → VerifiedKnowledge:
  - Подтверждение словарём (вес >= 0.9)
  ИЛИ
  - Подтверждение пользователем (вес = 1.0)
  ИЛИ
  - Confidence > 0.9 + Confirmations >= 10 + нет конфликта с ядром
```

### Переходы вниз

```text
VerifiedKnowledge → StableHypothesis:
  - Rejections > Confirmations * 0.3
  - И конфликт НЕ объясняется омонимией
    (контексты конфликтов пересекаются с контекстами подтверждений)

StableHypothesis → Hypothesis:
  - Rejections > Confirmations * 0.5
```

### Механизм при обнаружении конфликта

```text
Конфликт обнаружен
    │
    ├── Контексты конфликта отличаются от контекстов подтверждения?
    │   ├── Да → Создать новую интерпретацию (омонимия)
    │   │        Новая интерпретация = Hypothesis,
    │   │        проходит стандартный путь валидации
    │   │
    │   └── Нет → Понизить статус существующей (ошибка)
    │             Verified → Stable при Rejections > Confirmations * 0.3
    │             Stable → Hypothesis при Rejections > Confirmations * 0.5
```

Все пороги — стартовые значения. Калибруются на реальных данных начиная с v0.2.

## Стратегия тестирования

Тесты — обязательная часть проекта. Каждая версия выпускается только с пройденными тестами.

### Unit tests

Проверяют отдельно:
- определение типа фразы;
- определение намерения;
- морфологические признаки;
- переходы между статусами (вверх и вниз);
- создание альтернативной интерпретации при омонимии;
- работу весов источников.

### Integration tests

Проверяют полный путь:

## Система действий

Действие — это то, что агент выполняет после того как определил намерение. Действия делятся на два класса:

```text
EActionClass::Internal  — агент работает сам с собой (память, контекст)
EActionClass::External  — агент выдаёт результат пользователю (текст)
```

Это деление принципиально: External-действие всегда порождает `ResultText` в ответе. Internal-действие может не производить видимого текста — оно меняет внутреннее состояние агента.

---

### Каталог действий v0.1

#### External — текстовые действия

| ActionID | Что делает |
|---|---|
| `GET_DEFINITION` | Вернуть определение слова/понятия |
| `GET_WORD_FACT` | Рассказать факт о слове (происхождение, морфология) |
| `CHECK_TEXT_ERRORS` | Проверить текст на ошибки |
| `FIND_MEANING` | Найти значение слова |
| `ANSWER_ABILITY` | Ответить на вопрос «можешь ли ты X» |

#### Internal — внутренние процессы

| ActionID | Что делает |
|---|---|
| `STORE_FACT` | Сохранить утверждение в память фактов |
| `RETRIEVE_FROM_MEMORY` | Достать факт или контекст из памяти |
| `CHECK_MEMORY_LOAD` | Проверить объём оперативной памяти (HOT) |

В v0.1 реализуются: все External + `STORE_FACT`. Остальные Internal — с v0.3.

---

### Контракт действия

Каждое действие получает унифицированный `FActionRequest` и возвращает `FActionResult`.

```text
FActionRequest
├── ActionID       : EActionID    — что выполнить
├── EntityTarget   : FString      — объект действия (слово, фраза, текст)
└── Confidence     : float        — уверенность интерпретации [0..1]

FActionResult
├── bSuccess       : bool
├── ResultText     : FString      — для External-действий; пусто для Internal
├── FailReason     : EActionFailReason
└── DiagnosticNote : FString      — причина сбоя для логов
```

```text
EActionFailReason:
  None             — успех
  NotFound         — объект не найден в памяти / словаре
  NotSupported     — действие не реализовано в текущей версии
  LowConfidence    — уверенность ниже минимального порога
  InternalError    — ошибка выполнения
```

---

### Поведение при сбое

| FailReason | Поведение агента |
|---|---|
| `NotFound` | `bSuccess = false`, генератор формирует «не нашла» |
| `NotSupported` | `bSuccess = false`, генератор формирует «пока не умею» |
| `LowConfidence` | `bSuccess = false`, генератор добавляет низкий opener + уточняющий closer |
| `InternalError` | `bSuccess = false`, залогировать, генератор формирует нейтральный fallback |

Пустой ответ недопустим. Если действие не выполнилось по любой причине — генератор всегда строит ответ на основе `FailReason`.

---

### Реестр действий

Реестр — статическая таблица `TMap<EActionID, FActionHandler>`. Заполняется при старте агента.

```text
FActionRegistry
├── Register(EActionID, FActionHandler)  — регистрация обработчика
├── Execute(FActionRequest) → FActionResult  — выполнение
└── IsSupported(EActionID) → bool        — проверка наличия
```

Добавление нового действия в будущих версиях:
1. Добавить `ActionID` в `EActionID`.
2. Написать обработчик `FActionHandler`.
3. Зарегистрировать в реестре при старте.
4. Ни один существующий модуль не меняется.

Реестр не знает о разделении Internal/External — это семантическое свойство самого ActionID, а не инфраструктурное.

---

### Связь с pipeline

```text
Intent
  → ActionSelector (выбирает ActionID по Intent + PhraseType)
      → FActionRegistry::Execute(FActionRequest)
          → FActionResult
              → FResponseGenerator (формирует текст)
```

`ActionSelector` — простая таблица `Intent → ActionID` в v0.1. Логика усложняется с v0.3, когда появятся вложенные намерения.

## Известные риски

1. **Суффиксные правила без словаря быстро ломаются** на русском языке. Решение: словарь-ядро начиная с v0.2.
2. **Омонимия требует нескольких интерпретаций на слово.** Заложено в архитектуру с самого начала.
3. **Самообучение без валидации закрепляет мусор.** Решение: три состояния знания + жёсткий порог.
4. **Семантический граф на грязных данных = граф ошибок.** Решение: граф только после стабильных нижних слоёв (v0.5).
5. **Прагматика** («Тут холодно» = «Закрой окно») — самый сложный уровень. Откладывается.
6. **Понимание C++ и самомодификация** — отдельная ветка сложности, не часть языкового ядра.
7. **Покрытие словарём ~80%** — рабочая гипотеза, зависит от жанра текста. Калибруется на реальных данных.

## Dependency Policy / Политика зависимостей

Проект должен придерживаться следующих правил:

1. **Не добавлять внешнюю зависимость без причины.**  
   Каждая новая библиотека должна отвечать на вопрос:  
   _какую конкретную проблему она решает, которую нельзя разумно решить уже имеющимися средствами?_

2. **Разделять runtime и tooling зависимости.**  
   - runtime — нужны агенту во время работы;
   - tooling — нужны только для подготовки словарей, тестов и данных.

3. **Предпочитать встроенные возможности Unreal, если они достаточны.**  
   Это снижает сложность сборки и уменьшает риск несовместимости.

4. **Фиксировать версии критичных зависимостей.**  
   Особенно для:
   - Unreal Engine,
   - компилятора,
   - словарных дампов,
   - инструментов конвертации.

5. **Не превращать v0.1 в зависимый от тяжёлой инфраструктуры этап.**  
   Первая версия должна запускаться с минимальным набором внешних данных.
