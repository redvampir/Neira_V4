# Нейра — Саморазвивающийся символьный ИИ-агент

## О проекте

**Нейра** — экспериментальный ИИ-агент, который строится не на статистическом угадывании, а на явных правилах понимания, проверяемых гипотезах и управляемом накоплении знаний.

Проект создаётся как **исследовательская система**, а не как игровой NPC. Unreal Engine здесь используется как удобный фреймворк, среда интеграции и визуальный слой.

- **Язык ядра:** C++
- **Среда:** Unreal Engine
- **Визуальный слой:** Blueprints
- **Главный язык анализа:** русский
- **Второй язык:** английский

## Философия

- Агент **не угадывает**, а разбирает вход по правилам.
- Самообучение идёт через **гипотезы, подтверждения и пересмотр**, а не через слепую фиксацию выводов.
- Ошибки не должны цементироваться: знания могут не только усиливаться, но и **деградировать**.
- Омонимия и неоднозначность считаются **нормой языка**, а не ошибкой системы.
- Внешние LLM допустимы как вспомогательный инструмент обучения и проверки, но не как runtime-мозг.

## Scope и границы проекта

### Что входит в проект

- интерпретация естественного языка;
- морфологический и далее синтаксический разбор;
- хранение гипотез, знаний и контекстных интерпретаций;
- управляемая валидация;
- построение ответа на основе внутренних правил;
- интеграция логики с Unreal / Blueprint.

### Что **не** является целью на текущем этапе

- AGI / полноценный общий интеллект;
- автономная самомодификация кода без строгой верификации;
- свободное самообучение без внешнего контроля;
- продакшн-замена LLM;
- немедленный запуск сложного семантического графа;
- полный морфемный разбор русского языка в v0.1.

## Целевая архитектура

Это **долгосрочная карта**, а не порядок немедленной реализации.

```text
Слой 0: Лингвистическое ядро        — словари, таблицы, правила, признаки
Слой 1: Морфологический анализатор  — форма слова → лемма → часть речи
Слой 2: Синтаксический парсер       — структура предложения и связи
Слой 3: Семантический граф          — понятия и отношения
Слой 4: Механизм рассуждений        — вывод, выбор действий, память
Слой 5: Генератор ответа            — формирование ответа по правилам
```

### Оценка веса модели

| Компонент | Размер |
|-----------|--------|
| Слой 0: словари + правила | ~100–150 МБ |
| Слои 1–2: код + таблицы правил | ~10–20 МБ |
| Слой 3: семантический граф (старт) | ~10–50 МБ |
| **Итого на старте** | **~200–300 МБ** |
| Семантический граф (рост) | до гигабайтов |

## План сборки

### v0.1 — минимальный интерпретатор фраз

Без словарей, на паттернах и базовых правилах.

- тип фразы: вопрос / команда / утверждение / просьба;
- намерение: узнать / выполнить / проверить возможность / уточнить;
- выделение объекта действия;
- фиксация причины вывода и уровня уверенности;
- накопление гипотез без записи в жёсткие знания.

### v0.2 — лёгкая морфология

- словарь-ядро: 5 000–10 000 частотных слов;
- леммы и части речи;
- определение части речи: сначала по словарю, потом по признакам;
- базовая обработка омонимии через контекст;
- первые устойчивые правила перехода гипотез в знания.

### v0.3 — синтаксические связи

- кто → действие → объект → адресат;
- вложенные действия;
- различение прямого действия и проверки возможности;
- контекстный выбор интерпретации слова.

### v0.4 — управляемое самообучение

- гипотезы, стабильные гипотезы, проверенные знания;
- переходы вверх и вниз;
- внешняя валидация;
- защита от закрепления ложных выводов.

### v0.5 — семантический граф

- только после стабилизации нижних слоёв;
- связи между понятиями;
- уплотнение, дедупликация, проверка чистоты данных.

### Далёкий этап — понимание C++ и самомодификация

Это отдельный крупный подпроект: лексер → парсер → AST → генерация кода → runtime compilation → hot reload → верификация. Не является продолжением языкового ядра.

## Рабочий pipeline агента

Базовый цикл работы системы:

```text
Ввод
 → Нормализация строки
 → Токенизация
 → Анализ слов / гипотез
 → Выбор интерпретаций по контексту
 → Определение типа фразы
 → Определение намерения
 → Выбор действия
 → Выполнение действия
 → Формирование ответа
 → Обновление памяти и статусов гипотез
```

## Интерфейс морфология → синтаксис

### Принятое решение

Морфологический анализатор обрабатывает **все токены сразу** и передаёт синтаксическому парсеру **полный список** со всеми возможными интерпретациями.

Синтаксис работает с полной картиной предложения — это позволяет разрешать омонимию с учётом контекста всего предложения, а не только предшествующих слов.

```text
Токенизация
 → Морфология (все токены сразу)
    → [токен1: [интерп1, интерп2], токен2: [...], ...]
       → Синтаксис (полная картина)
          → выбор активной интерпретации по контексту
```

### Почему не пошагово (токен за токеном)

Русский язык имеет свободный порядок слов. Интерпретация слова в начале предложения часто определяется словами в конце:

```text
«Печь жарко топили» → «печь» = существительное (топили что? = печь)
«Печь пироги умела»  → «печь» = глагол (умела что? = печь)
```

При пошаговой обработке синтаксис получает `«печь»` раньше, чем видит `«топили»` или `«пироги»` — разрешить омонимию невозможно.

### Триггер для выбора подхода

| Условие | Подход |
|---|---|
| Язык с фиксированным порядком слов | Пошаговая обработка допустима |
| Язык со свободным порядком слов (русский) | Полная картина обязательна |
| Омонимия — ключевая задача системы | Полная картина обязательна |

Для Нейры: омонимия — центральная задача, русский — свободный порядок слов. Полная картина обязательна.

Базовая схема интерпретации:

```text
Фраза → Тип фразы → Намерение → Целевое действие → Объект действия
```

### Примеры

| Фраза | Тип | Намерение | Действие | Объект |
|---|---|---|---|---|
| Дай определение слова мама | команда | получить знание | найти определение | слово «мама» |
| Ты можешь нарисовать квадрат? | вопрос | проверить возможность | ответить о возможности | нарисовать квадрат |

## Типы памяти

Проекту нужна не одна общая «память», а несколько разных слоёв хранения.

### 1. Оперативная память

Хранит текущее состояние обработки:
- входную фразу;
- токены;
- промежуточные гипотезы;
- активное намерение;
- выбранное действие.

Очищается после каждого цикла обработки.

### 2. Лексическая память

Хранит данные о словах:
- формы;
- леммы;
- части речи;
- интерпретации;
- контекстные маркеры.

Основное хранилище — `TMap<FString, FWordEntry>`.

### 3. Память правил

Хранит:
- правила переходов между статусами;
- морфологические признаки;
- шаблоны фраз;
- веса источников;
- условия деградации знаний.

Загружается при старте, расширяется через самообучение (только после валидации).

### 4. Память фактов

Хранит интерпретированные утверждения и связи:
- «Кайл боится лабораторий»;
- «мама — существительное»;
- «дай — повелительная форма».

### 5. Долговременная память знаний

Хранит только устойчивые и проверенные структуры:
- VerifiedKnowledge;
- одобренные пользователем выводы;
- данные, подтверждённые сильными источниками.

Сериализуется на диск. Восстанавливается при запуске.

## Модель данных слова

### Принцип

Слово не равно одной трактовке. Одна форма может иметь несколько валидных интерпретаций. Это не исключение, а системное свойство русского языка.

### Двухуровневая схема

```text
FWordEntry (запись слова)
├── Text: FString                          — исходная форма
├── LemmaCandidates: TArray<FString>       — возможные леммы
├── Interpretations: TArray<FWordInterpretation>
└── CreatedAt / UpdatedAt                  — метаданные

FWordInterpretation (одна интерпретация)
├── WordType: EWordType                    — часть речи
├── State: EKnowledgeState                 — статус знания
├── Confidence: float                      — уверенность 0.0–1.0
├── Reason: FString                        — почему так решил
├── SourceWeight: float                    — вес источника
├── Confirmations: int32                   — число подтверждений
├── Rejections: int32                      — число опровержений
├── ContextMarkers: TArray<FString>        — когда эта интерпретация активна
└── History: TArray<FStateChange>          — история изменений статуса
```

### Пример

```text
Слово: «печь»
  Интерпретация 1:
    тип: существительное
    статус: VerifiedKnowledge
    уверенность: 0.95
    источник: словарь (вес 0.9)
    подтверждений: 34 / опровержений: 0
    контекст: после предлога, после прилагательного

  Интерпретация 2:
    тип: глагол
    статус: VerifiedKnowledge
    уверенность: 0.92
    источник: словарь + контекст (вес 0.9)
    подтверждений: 51 / опровержений: 0
    контекст: после модального глагола, перед сущ. в вин. падеже
```

Выбор активной интерпретации — задача отдельного механизма, привязанного к контексту (v0.3), а не морфологического слоя.

## Состояния знаний

```text
Hypothesis          — начальная догадка, может быть ошибочной
StableHypothesis    — подтверждена несколько раз, мало опровержений
VerifiedKnowledge   — проверена сильным источником
```

### Переходы вверх

```text
Hypothesis → StableHypothesis:
  - Confidence > 0.7
  - Confirmations >= 3
  - Rejections < Confirmations * 0.2

StableHypothesis → VerifiedKnowledge:
  - Подтверждение словарём (вес >= 0.9)
  ИЛИ
  - Подтверждение пользователем (вес = 1.0)
  ИЛИ
  - Confidence > 0.9 + Confirmations >= 10 + нет конфликта с ядром
```

### Переходы вниз

```text
VerifiedKnowledge → StableHypothesis:
  - Rejections > Confirmations * 0.3
  - И конфликт НЕ объясняется омонимией
    (контексты конфликтов пересекаются с контекстами подтверждений)

StableHypothesis → Hypothesis:
  - Rejections > Confirmations * 0.5
```

### Механизм при обнаружении конфликта

```text
Конфликт обнаружен
    │
    ├── Контексты конфликта отличаются от контекстов подтверждения?
    │   ├── Да → Создать новую интерпретацию (омонимия)
    │   │        Новая интерпретация = Hypothesis,
    │   │        проходит стандартный путь валидации
    │   │
    │   └── Нет → Понизить статус существующей (ошибка)
    │             Verified → Stable при Rejections > Confirmations * 0.3
    │             Stable → Hypothesis при Rejections > Confirmations * 0.5
```

Все пороги — стартовые значения. Калибруются на реальных данных начиная с v0.2.

## Разделение C++ / Blueprint

### C++

- логика языка;
- морфология;
- синтаксис;
- все типы памяти;
- гипотезы и статусы;
- правила и их переходы;
- JSON / файлы / словари;
- механизм действий;
- сериализация состояния.

### Blueprint

- UI;
- визуализация состояния агента;
- дебаг (экранный вывод, логи);
- прототипирование взаимодействий;
- отображение результатов анализа.

### Интеграция

```cpp
UCLASS(BlueprintType, Blueprintable, meta=(DisplayName="Русское имя"))
UFUNCTION(BlueprintCallable, meta=(DisplayName="Русское имя"))
UPROPERTY(BlueprintReadWrite, meta=(DisplayName="Русское имя"))
```

Agent → `UObject` или `GameInstanceSubsystem`
Memory → отдельный `UObject` на каждый тип памяти
Actions → `BlueprintCallable` методы

## Build / Run / Test

### Минимальные требования

- Unreal Engine: зафиксировать версию в проекте;
- C++ стандарт: зафиксировать явно (C++17 минимум);
- словарь-ядро подключается как внешний ресурс в `Data/Dictionaries/`;
- все тесты запускаются отдельно от Blueprints.

### Первый запуск v0.1

Цель:
- принять строку;
- определить тип фразы;
- определить намерение;
- вернуть parse-result;
- записать гипотезу в память.

### Первый запуск v0.2

Цель:
- подключить словарь-ядро;
- получить часть речи хотя бы для 80% частотных слов;
- сохранить несколько интерпретаций одного слова;
- протестировать омонимию на контрольных случаях.

## Стратегия тестирования

Тесты — обязательная часть проекта. Каждая версия выпускается только с пройденными тестами.

### Unit tests

Проверяют отдельно:
- определение типа фразы;
- определение намерения;
- морфологические признаки;
- переходы между статусами (вверх и вниз);
- создание альтернативной интерпретации при омонимии;
- работу весов источников.

### Integration tests

Проверяют полный путь:
- вход → разбор → действие → ответ → обновление памяти.

### Regression tests

Обязательные контрольные кейсы:

| Кейс | Что проверяет |
|------|---------------|
| «печь» | Омонимия: существительное / глагол |
| «стекло» | Омонимия: существительное / глагол |
| «простой» | Омонимия: прилагательное / существительное |
| «три» | Омонимия: числительное / глагол (повел. от «тереть») |
| «Ты можешь нарисовать квадрат?» | Вложенное намерение: вопрос о возможности ≠ команда |
| «Дай определение слова мама» | Команда + объект + вложенная конструкция |

## Критерии готовности v0.1

v0.1 считается рабочей, если система умеет:

- различать вопрос / команду / утверждение / просьбу;
- извлекать намерение из набора базовых шаблонов;
- выделять объект действия в простых фразах;
- сохранять гипотезу с причиной и уверенностью;
- не записывать вывод в VerifiedKnowledge без валидации;
- проходить все unit-тесты для типов фраз и намерений.

## Критерии готовности v0.2

v0.2 считается рабочей, если система умеет:

- использовать словарь-ядро (5 000–10 000 слов);
- определять часть речи хотя бы у 80% частотных слов из словаря;
- хранить несколько интерпретаций одного слова;
- отличать ошибку от омонимии в простых случаях;
- повышать статус гипотезы при подтверждении;
- понижать статус знания при реальном конфликте;
- проходить все регрессионные тесты на омонимию.

## Источники данных и лицензии

Раздел предварительный. Фиксируется формально с v0.2.

| Источник | Назначение | Лицензия | Формат |
|----------|-----------|----------|--------|
| OpenCorpora | Русская морфология, леммы | CC BY-SA | XML → внутренний формат |
| WordNet | Английская морфология | Princeton WordNet License | DB → внутренний формат |
| Собственные таблицы | Правила, шаблоны, признаки | — | JSON / бинарный |
| Пользовательские данные | Подтверждения, факты | — | Внутренний формат |

Словари хранятся в `Data/Dictionaries/`. Полные дампы в репозиторий не включаются, если лицензия запрещает. Вместо этого — скрипт загрузки и конвертации.

## Известные риски

1. **Суффиксные правила без словаря быстро ломаются** на русском языке. Решение: словарь-ядро начиная с v0.2.
2. **Омонимия требует нескольких интерпретаций на слово.** Заложено в архитектуру с самого начала.
3. **Самообучение без валидации закрепляет мусор.** Решение: три состояния знания + жёсткий порог.
4. **Семантический граф на грязных данных = граф ошибок.** Решение: граф только после стабильных нижних слоёв (v0.5).
5. **Прагматика** («Тут холодно» = «Закрой окно») — самый сложный уровень. Откладывается.
6. **Понимание C++ и самомодификация** — отдельная ветка сложности, не часть языкового ядра.
7. **Покрытие словарём ~80%** — рабочая гипотеза, зависит от жанра текста. Калибруется на реальных данных.

## Dependency Policy / Политика зависимостей

Проект должен придерживаться следующих правил:

1. **Не добавлять внешнюю зависимость без причины.**  
   Каждая новая библиотека должна отвечать на вопрос:  
   _какую конкретную проблему она решает, которую нельзя разумно решить уже имеющимися средствами?_

2. **Разделять runtime и tooling зависимости.**  
   - runtime — нужны агенту во время работы;
   - tooling — нужны только для подготовки словарей, тестов и данных.

3. **Предпочитать встроенные возможности Unreal, если они достаточны.**  
   Это снижает сложность сборки и уменьшает риск несовместимости.

4. **Фиксировать версии критичных зависимостей.**  
   Особенно для:
   - Unreal Engine,
   - компилятора,
   - словарных дампов,
   - инструментов конвертации.

5. **Не превращать v0.1 в зависимый от тяжёлой инфраструктуры этап.**  
   Первая версия должна запускаться с минимальным набором внешних данных.
