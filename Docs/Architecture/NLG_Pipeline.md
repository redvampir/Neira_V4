# Архитектура NLG-конструктора Нейры (2026-03-20)

## Суть идеи

LLM строит текст токен за токеном, угадывая каждый следующий статистически.
Нейра строит текст **концепт за концептом**, выводя каждый следующий из смысла.

Это зеркальное отражение NLU: если NLU разбирает текст в семантический фрейм,
то NLG собирает семантический фрейм в текст.

```
NLU: "Кот является животным" ──→ { Subject=кот, Pred=является, Object=животное }
NLG: { Subject=кот, Pred=является, Object=животное } ──→ "Кот является животным"
                                                      или "Кот — это животное"
                                                      или "Кот относится к животным"
```

Одинаковый смысл → разные поверхностные формы. Выбор формы — детерминирован личностью и контекстом.

---

## Конвейер NLG

```
FBeliefEngine
      │
      ▼
FResponseSemantics   ← "что сказать" (смысл, уверенность, иллокуция)
      │
      ▼
FSentencePlanner     ← "как структурировать" (синтаксическая стратегия)
      │
      ▼
FAbstractSyntaxTree  ← дерево: субъект + предикат + зависимости
      │
      ▼
FMorphRealizer       ← "конкретные слова" (падеж, число, род, вид глагола)
      │
      ▼
FString              ← итоговый текст ответа
```

---

## Компонент 1: FResponseSemantics

**Что это:** декларация смысла ответа. Не строка — структура.

```cpp
struct FResponseSemantics
{
    EIllocution    Illocution;    // Assert, Question, Hedge, Refuse, Acknowledge
    EConfidence    Confidence;    // Verified, Inferred, Uncertain, Unknown
    FProposition   Proposition;   // { Subject, Predicate, Object, Modifiers[] }
    FPersonality   Personality;   // { Tone, Register, AddressStyle }
    FString        TopicContext;  // для связности с предыдущим высказыванием
};

struct FProposition
{
    FString Subject;
    FString Predicate;    // лемма глагола или связки
    FString Object;
    TArray<FString> Modifiers;
    bool bIsNegated;
};
```

Именно здесь находится "то, что Нейра хочет сказать" — до того, как это стало русским текстом.

---

## Компонент 2: FSentencePlanner

**Что это:** выбор синтаксической стратегии для данного смысла.

Для одного смысла существует несколько корректных синтаксических структур.
Плановщик выбирает одну — детерминировано по (Personality + контекст + история выборов).

### Стратегии для Assert + Verified:

| Стратегия | Пример | Когда предпочтительна |
|---|---|---|
| `Copula_Является` | "Кот является животным" | Формальный тон |
| `Copula_Dash` | "Кот — это животное" | Нейтральный тон |
| `Copula_Relative` | "Кот относится к животным" | Технический контекст |
| `Existential` | "Среди животных — кот" | Акцент на категории |

### Стратегии для Hedge + Uncertain:

| Стратегия | Пример |
|---|---|
| `Hedge_KazhetJa` | "Кажется, кот является животным" |
| `Hedge_Vidimo` | "По всей видимости, кот — животное" |
| `Hedge_Pologaju` | "Полагаю, что кот относится к животным" |
| `Hedge_NeTochno` | "Точно не уверена, но кот — скорее всего животное" |

**Источник разнообразия без случайности:**
```
StrategyIndex = Hash(LastUsedStrategy, TopicContext, SessionCount) % AvailableStrategies.Num()
```
Детерминировано, но не монотонно.

## Компонент 3: FMorphRealizer

**Что это:** согласование слов по правилам русской грамматики.

Принимает абстрактное дерево вида `{Subject(nom) + Predicate(3sg,pres) + Object(inst)}`,
возвращает конкретные словоформы.

**Ключевые задачи для русского:**

| Задача | Пример |
|---|---|
| Падежное управление | "являться" + объект → инструментальный падеж |
| Числовое согласование | субъект мн.ч. → глагол мн.ч. |
| Родовое согласование | "белый/белая/белое кот/кошка" |
| Вид глагола | "знать" (impf) vs "узнать" (pf) по контексту |
| Одушевлённость | вин.пад. одушевлённых = род.пад. |

**Связь с FMorphAnalyzer:**
FMorphAnalyzer уже умеет разбирать словоформу → лемму + теги.
FMorphRealizer — обратная задача: лемма + теги → словоформа.
Это означает: словарные данные уже есть, нужна обратная функция поиска.

## Инструменты реализации

### Фаза 1: SimpleNLG-RU (рекомендован для старта)

```
Язык:     Java/C++ порт
Лицензия: Apache 2.0
Что даёт: согласование форм + базовые синтаксические стратегии для русского
```

Позволяет описать предложение как структуру (не строку) и получить корректно
согласованный русский текст.

### Фаза 2: Grammatical Framework (GF)

```
Язык:     Haskell + C++ runtime (libpgf)
Лицензия: LGPL
Что даёт: полноценная грамматика русского; двунаправленный (parse + generate)
```

GF позволяет описать смысл в **абстрактном синтаксисе**, независимом от языка:
```
mkS (mkNP кот) (mkVP являться (mkNP животное))
→ "Кот является животным"  (RU)
→ "A cat is an animal"     (EN)  — при необходимости
```

---

## Чем это отличается от LLM

```
LLM:
  кот → является → животным → [0.72 вероятность] → следующий токен...
  Нейра не знает ПОЧЕМУ выбрано слово. Нельзя проверить.

Нейра NLG:
  FProposition{кот, являться, животное}
    → Copula_Является стратегия
    → Subject: кот [nom, masc, sg, anim] ← из FMorphRealizer
    → Pred:    является [3sg, pres, impf] ← управление глагола
    → Object:  животным [inst, neut, sg] ← падежное управление "являться"
    → "Кот является животным."
  Каждый шаг — проверяем, объясняем, воспроизводим.
```

## Связь с существующей архитектурой

| Существующий компонент | Роль в NLG |
|---|---|
| `FBeliefEngine` | Источник `FResponseSemantics` |
| `FMorphAnalyzer` | Словарная база для `FMorphRealizer` (обратная функция) |
| `FPersonalityProfile` | Вход для `FSentencePlanner` (выбор стратегии) |
| `FResponseGenerator` | Оркестратор всего NLG-конвейера |
| `FSemanticGraph` | Обогащение `FProposition` (синонимы, гиперонимы) |
