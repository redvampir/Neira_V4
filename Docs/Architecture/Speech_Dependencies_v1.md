# Speech Dependencies v1

## Цель
Выбрать минимально-рисковый набор speech-зависимостей (ASR/TTS) для RU-first продукта с возможностью деградации без поломки публичного API Neira.

## Допущения
- Под «speech» в этом документе подразумеваются две подсистемы: **ASR** (speech-to-text) и **TTS** (text-to-speech).
- Приоритет: **качество RU + управляемая отказоустойчивость** над абсолютной минимизацией vendor lock-in.
- Оценки качества/латентности даны как архитектурные ориентиры для v1 и должны быть подтверждены бенчмарком на целевых устройствах.

## Выбор v1

### Primary stack (основной)
1. **Yandex SpeechKit (Cloud STT + Cloud TTS)** как единый cloud-провайдер для RU-first сценариев.
2. Причина: наилучший баланс RU-качества, latency и простоты эксплуатации в первой версии.

### Fallback stack (отказоустойчивость)
1. **ASR fallback:** Vosk (локально, офлайн).
2. **TTS fallback:** RHVoice (локально, офлайн).
3. Роль fallback-слоя: продолжать базовый voice-контур при потере сети, недоступности cloud API или превышении budget по стоимости/квотам.

## Почему выбран именно этот набор

1. **RU-first качество:** primary ориентирован на высокое качество русской речи в проде.
2. **Операционная устойчивость:** fallback не зависит от сети и внешнего SLA, снижает риск «полного немого отказа».
3. **Контроль рисков лицензирования:** cloud и offline зависимости разведены по слоям и могут заменяться независимо.
4. **Инкрементальная интеграция:** сначала один primary-провайдер, затем безопасное подключение fallback-провайдеров без смены публичных контрактов.

## Ограничения v1

1. **Vendor lock-in** на primary (cloud): требуются адаптеры и контрактный слой, чтобы избежать жёсткой привязки к API провайдера.
2. **Качество offline ниже** cloud в общих сценариях (особенно шум, дальняя речь, редкие доменные термины).
3. **Ресурсоёмкость offline ASR:** модели и CPU-нагрузка влияют на latency и FPS в UE.
4. **Комплаенс/данные:** cloud-контур требует формализованной privacy/security-политики (PII, retention, deletion, audit).
5. **Оценки в таблице предварительные:** обязательны бенчмарки перед production freeze.

## План миграции без ломки API

### 1) Зафиксировать стабильные интерфейсы (без зависимости от vendor SDK)
- `ISpeechRecognizer` (ASR): `StartSession()`, `PushAudioChunk()`, `StopAndFinalize()`, `Cancel()`.
- `ISpeechSynthesizer` (TTS): `Synthesize(text, voice, format)`.
- `FSpeechRequest/FSpeechResult` как versioned DTO (`schema_version`).

### 2) Ввести provider-адаптеры
- `FSpeechProviderYandex` (primary).
- `FSpeechProviderVosk` (fallback ASR).
- `FSpeechProviderRHVoice` (fallback TTS).
- Переключение через конфиг и feature flags, не через изменения сигнатур API.

### 3) Реализовать политику деградации
- `Primary -> Fallback` при: timeout, 5xx/429, network unavailable, quota exceeded.
- Circuit breaker + backoff + health checks.
- Явный `ProviderUsed` в telemetry/diagnostics.

### 4) Сохранить совместимость контрактов
- Добавлять только backward-compatible поля в ответы.
- Не удалять существующие enum-значения; deprecated-only policy до v2.
- Версионировать контракты (`v1`, `v1.1`) и поддерживать минимум 2 соседние версии.

### 5) План выпуска
- Phase A: primary-only (dark launch fallback).
- Phase B: fallback для 1–5% сессий (canary).
- Phase C: full rollout с SLO-алертами.

## Definition of Done для этого решения
1. Есть реализация интерфейсного слоя ASR/TTS без прямой утечки vendor-типов в публичный API.
2. Переключение primary/fallback управляется конфигом без перекомпиляции.
3. Для отказа primary есть автодеградация в fallback с диагностикой причины.
4. Подготовлен benchmark-пакет (WER/RTF/latency/cost) для RU-набора фраз.
5. Документирована privacy/security-обвязка для cloud speech-потока.
