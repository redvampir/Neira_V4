# Критические незакрытые моменты архитектуры Neira (на 2026-03-19)

## Контекст
Документ фиксирует архитектурные пробелы, которые в текущем состоянии проекта остаются незакрытыми и блокируют безопасный переход от «дизайна» к рабочей системе v0.1–v0.2.

## 1) Не зафиксированы машинно-проверяемые контракты модулей
**Проблема:** в README есть концептуальные примеры входа/выхода, но нет формального API-контракта (схемы, версионирования, обязательных/необязательных полей, кодов ошибок).

**Риск:** несовместимые изменения между модулями (Tokenizer/Morph/Syntax/Reasoning) будут обнаруживаться поздно, уже на интеграции.

**Что закрыть:**
- JSON Schema (или аналог) для промежуточных структур (`FPhraseParseResult`, payload морфологии, статусные события памяти).
- Версионирование контракта (`schema_version`) + стратегия backward compatibility.
- Нормализованный каталог ошибок (коды + условия + recovery).

## 2) ✅ ЗАКРЫТО Модель согласованности памяти и транзакционность обновлений
**Решение (2026-03-19):** Реализован `FHypothesisEvent` (append-only event log) в `FHypothesisStore`.

**Что сделано:**
- Каждый успешный переход состояния (Store/Confirm/Verify/MarkConflicted) пишет запись в `EventLog`.
- Неудачные операции (Verify из Pending, Confirm невалидного ID) НЕ пишут в лог — нет «тихой» порчи.
- `ClearEventLog()` для сброса лога между сессиями.
- 5 тестов EventLog + 2 negative теста подтверждают корректность.

**Что остаётся:**
- Replay/rollback сценарии — для v0.4 `Downgrade()` (Verified → Conflicted → Pending).
- Идемпотентность повторной обработки — не специфицирована.

## 3) Нет чёткой стратегии разрешения конфликтов источников и дрейфа порогов
**Проблема:** заданы стартовые веса источников и пороги, но нет процедуры их калибровки и контроля деградации качества после переобучения.

**Риск:** система будет «плыть» от ручных корректировок и случайного шума, особенно в самообучении.

**Что закрыть:**
- Offline-калибровка порогов на фиксированных датасетах.
- Drift-мониторинг (качество по регрессионным наборам во времени).
- Политика «заморозки» правил в релизной ветке и controlled rollout новых весов.

## 4) ✅ ЗАКРЫТО (базовый уровень) Explainability/audit trail
**Решение (2026-03-19):** Реализован `DecisionTrace` в `FIntentResult` + EventLog в FHypothesisStore.

**Что сделано:**
- `FIntentResult::DecisionTrace` (FString) содержит имя сработавшего правила:
  - `"Frame.AbilityCheck"` — фреймовый путь, проверка способности
  - `"Frame.QuestionWithObject"` — фреймовый путь, вопрос с объектом
  - `"Pattern:что такое"` — паттерновый fallback
  - `"Fallback:Unknown"` — ни один путь не дал результата
  - `"EmptyInput"` — пустой вход
- `FHypothesisEvent::MethodName` + `Reason` фиксируют причину каждого перехода состояния.
- Тесты: DecisionTrace_NotEmpty, DecisionTrace_ContainsFrame.

**Что остаётся (v0.4):**
- Полный forensic dump: альтернативы + их веса + контекст фрейма.
- Единый trace-id на весь pipeline (от FPhraseClassifier до FActionRegistry).
- Экспортируемый forensic-режим для регрессионного анализа.

## 5) Не формализована граница ответственности C++ vs Blueprint на уровне SLA
**Проблема:** функциональное разделение есть, но нет нефункциональных контрактов (latency budget, потокобезопасность, допустимые side effects).

**Риск:** утечки логики в Blueprint и деградация производительности/детерминизма под нагрузкой.

**Что закрыть:**
- SLA по времени на этапы pipeline.
- Политика threading/async и владения памятью UObject.
- Запрет критической бизнес-логики в Blueprint (только orchestration/UI).

## 6) Не определён security/privacy baseline для пользовательских данных
**Проблема:** заявлено хранение пользовательских подтверждений и фактов, но нет модели доступа, удаления, анонимизации, аудита.

**Риск:** юридические и репутационные риски при накоплении персональных данных и долгоживущей памяти.

**Что закрыть:**
- Data classification (PII/non-PII).
- Retention policy (TTL, правила удаления, экспорт/удаление по запросу).
- Минимальные требования к шифрованию at rest/in transit.

## 7) Не описана стратегия эволюции к v0.3+ без «архитектурного долга на швах»
**Проблема:** roadmap есть, но не определены migration path и точки стабилизации интерфейсов перед добавлением синтаксиса/семантики.

**Риск:** повторная перепрошивка базовых структур при каждом этапе (дорого и рискованно).

**Что закрыть:**
- Architectural Decision Records (ADR) для ключевых развилок.
- Definition of Done для каждого слоя с «заморозкой» интерфейсов.
- Migration playbook (скрипты миграции памяти/правил/словаря).

## Приоритет закрытия (рекомендуемый)
1. Контракты модулей + каталог ошибок.
2. Транзакционность/согласованность памяти.
3. Explainability/audit trail.
4. Drift-контроль и калибровка порогов.
5. Security/privacy baseline.
6. SLA C++/Blueprint.
7. Migration path v0.3+.

## Критерии приёмки «архитектура готова к реализации v0.1/v0.2»
- Для каждого шага pipeline есть формальный контракт и negative tests.
- Любое обновление памяти воспроизводимо через event log (replay) и не оставляет частичных состояний.
- Любой parse-result объясним до уровня правил/весов/контекста.
- Любое изменение порогов проходит через регрессионный gate.
- Пользовательские данные проходят по базовым privacy-политикам и управляемому удалению.

## Что могу закрыть самостоятельно vs где нужны уточнения владельца

### Могу закрыть самостоятельно (без блокеров)
1. **Контракты модулей (черновик v1):**
   - подготовить JSON Schema для ключевых промежуточных структур;
   - завести `schema_version` и правила эволюции.
2. **Каталог ошибок:**
   - предложить единый набор error-кодов по pipeline;
   - добавить negative cases для v0.1/v0.2.
3. **Explainability-каркас:**
   - формализовать trace-формат (trace-id, шаг, правило, вес, альтернатива);
   - подготовить шаблон forensic dump.
4. **ADR/DoD-шаблоны:**
   - добавить шаблоны архитектурных решений и Definition of Done по слоям.
5. **Базовый regression gate для порогов:**
   - оформить процедуру сравнения качества «до/после» на фиксированном наборе кейсов.

### Нужны уточнения от тебя (owner decisions)
1. **Политика данных и приватности:**
   - какие классы данных считаем чувствительными (PII),
   - какие сроки хранения и обязательные сценарии удаления.
2. **Источник «истины» при конфликте:**
   - приоритет словаря vs пользовательского подтверждения vs ручной правки команды.
3. **Целевые SLA:**
   - допустимая задержка ответа на фразу (p50/p95),
   - ограничения по памяти для v0.1/v0.2.
4. **Границы самообучения в релизе:**
   - разрешаем ли онлайн-обновление весов,
   - или только через offline-калибровку и релизный rollout.
5. **Платформенный фокус ближайшего этапа:**
   - только RU-контур или сразу задел под EN-контур.

## Минимальный пакет уточнений (чтобы двигаться без пауз)
- retention policy (срок хранения пользовательских данных),
- source-of-truth при конфликте интерпретаций,
- latency budget p95 для одного pipeline-цикла.

## Принятые owner-решения (зафиксировано)

Ниже — фиксация твоих ответов, чтобы команда могла работать без повторных согласований:

1. **Чувствительные данные:** считаем чувствительными прежде всего **личные данные**.
2. **Источник «истины» при конфликтах:** финальное решение по приоритетам принимает **команда** (через согласованный процесс/ADR).
3. **Производительность/SLA:** целевые задержки **допустимы**, но подстраиваются под фактические ограничения железа.
4. **Самообучение в релизе:** **разрешено** для простых данных и задач сверки информации.
5. **Языковый контур:** архитектурный задел делаем **и под английский** (EN), не только RU.

## Операционализация решений (чтобы это можно было проверять)

### A) Данные и приватность
- Ввести минимум два класса: `PII` и `NON_PII`.
- Для `PII` включить отдельные правила retention и удаления по запросу.

### B) Командный процесс при конфликтах интерпретаций
- Любое изменение приоритетов источников фиксировать в ADR с датой, владельцем и причиной.
- До утверждения ADR действует последняя стабильная матрица приоритетов.

### C) SLA с поправкой на железо
- Ввести профильные режимы: `baseline`, `limited_hardware`.
- При деградации ресурсов переключать профиль без смены логики интерпретации.

### D) Рамки разрешённого самообучения
- Разрешить online-апдейты только для low-risk классов (простые данные/сверка).
- Для остальных классов — только offline-калибровка + регрессионный gate.

### E) RU+EN архитектурный задел
- В интерфейсах и схемах не хардкодить предположения «только русский».
- Добавить language-aware поля и тестовые кейсы RU/EN в roadmap.

## Критерии приёмки по этим решениям
- Есть формальная классификация `PII/NON_PII` и тесты маршрутизации хранения.
- Есть ADR-шаблон и минимум один ADR по source-of-truth.
- Есть два профильных performance-конфига и проверка переключения.
- Есть whitelist сценариев online-learning для low-risk данных.
- Есть минимум по 1 regression-кейсу для RU и EN в одинаковом формате контракта.
