# ADR 0003: Построение ответов через семантический конструктор (NLG), а не шаблоны

- Статус: accepted
- Дата: 2026-03-20
- Владелец: Team
- Заменяет: -
- Заменён: -

## Контекст

Генератор ответов (`FResponseGenerator`) в текущей архитектуре опирается на шаблоны строк
с подстановкой (`"Я знаю что {X} является {Y}"`). Это создаёт три проблемы:

1. **Ограниченное разнообразие**: одна структура предложения на каждый тип ответа.
2. **Жёсткая связка формы и содержания**: изменение смысла требует переписывания шаблона.
3. **Отсутствие морфологического согласования**: падежи, род, число приходится встраивать
   в шаблон вручную, что приводит к ошибкам в русском языке.

При этом LLM генерируют текст пословно, но статистически — угадывая следующий токен.
Принципиальный вопрос: можно ли строить текст пословно, но **по значению и семантике**,
без статистического угадывания?

**Ответ: да. Это называется NLG (Natural Language Generation).**

## Решение

Принять архитектуру **семантического конструктора ответов** — двустороннего символического
конвейера: NLU разбирает вход в смысл, NLG собирает смысл в текст.

```
[NLU] Текст → SemanticFrame → FBeliefEngine → FResponseSemantics
[NLG] FResponseSemantics → FSentencePlanner → FMorphRealizer → Текст ответа
```

### Как работает конструктор (в отличие от LLM)

| | LLM | Нейра NLG |
|---|---|---|
| Единица генерации | Токен | Смысловой узел (концепт + отношение) |
| Основание выбора | P(токен \| контекст) — статистика | SemanticFrame + синтаксические правила |
| Согласование слов | Неявное, из весов | Явные правила русской морфологии |
| Разнообразие | Из случайности (temperature) | Из множества синтаксических стратегий на один смысл |
| Детерминизм | Нет (при temperature > 0) | Да (при фиксированных входах) |

### Компоненты конструктора

**1. FResponseSemantics** — что сказать (выход BeliefEngine):
```
{
  Illocution:   Assert | Question | Hedge | Refuse
  Confidence:   Verified | Inferred | Uncertain | Unknown
  Proposition:  { Subject, Predicate, Object, Modifiers }
  Personality:  { Tone, Register, AddressStyle }
}
```

**2. FSentencePlanner** — как структурировать предложение:
- Выбирает синтаксическую стратегию для данного смысла
- "Кот является животным" / "Кот — это животное" / "Кот относится к животным"
- Выбор зависит от: Tone (формальный/разговорный), Register, предыдущего предложения

**3. FMorphRealizer** — согласование слов:
- Подбирает падеж, род, число, вид глагола по синтаксическому дереву
- Опирается на FMorphAnalyzer (уже реализован) — реализация это **обращение** морфоанализа
- Пример: Subject=nom, Object=inst при предикате "являться"

### Источник разнообразия

Разнообразие достигается без случайности:
- Для каждого типа `Illocution × Confidence` — несколько синтаксических стратегий
- `Personality.Tone` определяет ранжирование стратегий
- Выбор стратегии: детерминирован по (смысл + личность + последнее использование)
  → не повторяет одну и ту же конструкцию подряд

### Инструменты реализации

**Вариант A: Grammatical Framework (GF)** — рекомендован для фазы 2
- C++ библиотека + грамматика русского языка
- Описываешь смысл в абстрактном синтаксисе → получаешь русское предложение
- Двунаправленный: тот же инструмент для NLU и NLG
- Лицензия: LGPL

**Вариант B: SimpleNLG-RU** — рекомендован для фазы 1 (легче)
- Порт SimpleNLG на русский язык
- Реализует согласование рода/числа/падежа
- Легче интегрировать чем GF

**Вариант C: Собственный FMorphRealizer** — минимальный вариант
- Использует уже реализованный FMorphAnalyzer в обратном направлении
- Только согласование; синтаксические стратегии кодируются вручную
- Нет внешних зависимостей

## Альтернативы

1. **Шаблоны с вариациями** — 20-50 шаблонов на тип ответа. Проще реализовать, но форма
   по-прежнему отвязана от смысла. Принято как **временное решение до реализации NLG**.
2. **Локальная LLM для генерации** — пробовалось в аналогичном проекте, результат неудовлетворительный
   (высокие вычислительные требования, галлюцинации, непредсказуемость). **Отклонено.**
3. **Нейросетевой NLG (seq2seq)** — нарушает инвариант SBA (ADR-0002). **Отклонено.**

## Последствия

- Плюсы:
  - Нейра "понимает что говорит" — форма ответа выводится из смысла, а не наклеивается на него
  - Грамматически корректный русский без ручной правки падежей
  - Разнообразие без случайности: детерминировано, но не монотонно
  - NLU и NLG используют одно семантическое представление — единая модель языка

- Минусы/риски:
  - Значительный объём реализации (особенно FMorphRealizer для русского)
  - GF требует освоения нового инструмента
  - Фаза 1 (шаблоны) временно нарушает принцип "форма из смысла"

## Фазовый план

| Фаза | Решение | Когда |
|---|---|---|
| Текущая | Шаблоны с вариациями (20-50 вариантов) | v0.4–v0.5 |
| Фаза 1 NLG | SimpleNLG-RU: согласование + базовые стратегии | v0.5–v0.6 |
| Фаза 2 NLG | Grammatical Framework: полноценный конструктор | v0.7+ |

## Rollback / миграция

`FResponseGenerator` сохраняет интерфейс (`FResponseSemantics → FString`).
Переход SimpleNLG → GF или откат к шаблонам меняет только реализацию, не интерфейс.
