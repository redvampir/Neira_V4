# ADR-0001: Приоритет источников (Source of Truth Priority)

- Статус: Accepted
- Дата: 2026-03-19
- Владелец: Architecture Team
- Заменяет: —
- Заменён чем: —

## Контекст
В системе есть несколько источников интерпретаций и фактов: пользовательские подтверждения, ручные правки команды, словарь и алгоритмические выводы.

Без формального порядка при конфликте источников возникают:
- непредсказуемое поведение на интеграции;
- дрейф качества после частичных правок;
- сложность аудита и объяснения, почему принят конкретный результат.

## Решение
1. **Владение политикой приоритетов**: финальное решение о матрице приоритетов принимает команда через ADR-процесс.
2. **Базовая матрица (до замены новым ADR)**:
   - `team_manual_override` (явно утверждённая правка команды);
   - `user_confirmed_fact` (подтверждённый пользователем факт);
   - `curated_dictionary` (кураторский словарь);
   - `model_inference` (алгоритмический вывод).
3. Любое изменение матрицы приоритетов считается архитектурным изменением и фиксируется новым ADR, который при необходимости заменяет этот.
4. До принятия нового ADR действует последняя версия со статусом `Accepted`.

## Альтернативы
1. **Приоритет «последняя запись выигрывает»** — быстро, но приводит к шуму и нестабильности.
2. **Полностью ручное разрешение каждого конфликта** — высокое качество, но не масштабируется для v0.1/v0.2.
3. **Приоритет только по confidence score** — удобно для модели, но не покрывает бизнес-правила и owner-решения.

## Последствия
### Плюсы
- Конфликт-резолюция становится предсказуемой и аудируемой.
- Появляется единая точка управления изменениями (ADR).
- Снижается риск скрытого дрейфа от «тихих» правок.

### Компромиссы и риски
- ADR-процесс замедляет изменение приоритетов.
- Нужна дисциплина ведения статусов (`Accepted`/`Superseded`).

## Rollback
Триггеры отката:
- рост регрессий на фиксированных тест-наборах после смены приоритетов;
- обнаружение систематического конфликта с бизнес-правилами.

Шаги:
1. Присвоить новому ADR статус `Superseded`.
2. Вернуть применение предыдущей матрицы приоритетов со статусом `Accepted`.
3. Повторно прогнать регрессионные проверки конфликт-резолюции.

Ожидаемое состояние после отката:
- система снова использует последнюю стабильную матрицу приоритетов;
- трассировка решений указывает на возвращённые правила приоритета.
