# Agent Playbook

> Цель: дать каждому агенту единый, воспроизводимый и проверяемый способ вести сессию от intake до handoff, чтобы не терять контекст и не накапливать архитектурный долг.

Дата фиксации: **2026-03-19**

---

## 1) Intake задачи (что читать первым)

Перед любыми изменениями агент проходит обязательный порядок чтения:

1. `README.md` — актуальные цели, границы scope, архитектурные слои и терминология.
2. `Docs/Roadmap/Agent_Roadmap.md` — текущий статус работ, приоритетная очередь, DoD и правила обновления после сессии.
3. `Docs/Architecture/Critical_Open_Architecture_Gaps.md` — критические риски и пробелы, которые нельзя ухудшать правкой.

Проверка intake (минимум):
- зафиксировано, **какой пункт roadmap** закрывается этой сессией;
- зафиксировано, **какой критический gap** затрагивается или сознательно не затрагивается;
- зафиксированы входные ограничения (время, зависимости, отсутствие данных).

---

## 2) Правила исполнения

### 2.1 Что считается «изменением»
Изменением считается любой артефакт в репозитории, который влияет на проект:
- код;
- спецификация/контракт;
- тест;
- документация;
- конфигурация/политика.

Не считается полноценным результатом:
- только устное описание без артефакта в репозитории;
- правка без связи с roadmap/риском/критерием приёмки.

### 2.2 Как описывать риски
Для каждой существенной правки фиксировать минимум:
- **Риск:** что может сломаться (совместимость, данные, производительность, безопасность).
- **Влияние:** какой модуль/слой затрагивается.
- **Снижение риска:** тест, флаг, rollback-путь, ограничение области применения.

Рекомендуемый формат:
- `Risk: <кратко>`
- `Impact: <модуль/интерфейс>`
- `Mitigation: <проверка/ограничение>`

### 2.3 Как фиксировать допущения
Если данных не хватает, допущения обязательны к явной маркировке:
- `Assumption: <что предполагаем>`
- `Reason: <почему без этого нельзя двигаться>`
- `Validation path: <как и когда проверить>`

Правило: каждое допущение должно иметь путь валидации (кто/чем/когда подтверждает или опровергает).

---

## 3) Правила завершения сессии

Сессия считается закрытой только если выполнены все пункты:

1. **Обновлён roadmap** (`Docs/Roadmap/Agent_Roadmap.md`) по правилам раздела обновления после сессии.
2. **Сделан handoff**: что сделано, что осталось, где блокеры.
3. **Есть ссылки на артефакты**: файлы, тесты, ADR, схемы, отчёты.
4. **Проверены критерии done** для задачи (локальные + из roadmap).

Минимальный handoff-блок:
- `Done:` фактический результат;
- `Next:` следующий минимальный шаг;
- `Blockers:` что мешает или `none`;
- `Artifacts:` список путей в репозитории.

---

## 4) Шаблон отчёта агента (единый формат, 5–7 пунктов)

Использовать один и тот же формат во всех сессиях:

1. **Session Context** — дата, цель, связанный пункт roadmap.
2. **Scope** — что включено/исключено в этой сессии.
3. **Changes** — список внесённых изменений (по файлам).
4. **Risks & Assumptions** — риски, допущения, меры снижения.
5. **Validation** — какие проверки запущены и результат.
6. **Roadmap Update** — что обновлено в roadmap после сессии.
7. **Handoff** — done/next/blockers/artifacts.

Шаблон (копировать в отчёты):

```md
## Agent Session Report

1) Session Context
- Date:
- Goal:
- Roadmap link/item:

2) Scope
- In scope:
- Out of scope:

3) Changes
- <file/path>: <что изменено>

4) Risks & Assumptions
- Risk:
- Impact:
- Mitigation:
- Assumption:
- Validation path:

5) Validation
- Commands/checks:
- Result:

6) Roadmap Update
- Updated section:
- New status/fact:

7) Handoff
- Done:
- Next:
- Blockers:
- Artifacts:
```

## Критерий применения playbook

Любая задача агента (кодовая или документационная) должна проходить через этот playbook как baseline процесса.
