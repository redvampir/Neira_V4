# Agent Playbook (на 2026-03-19)

Цель: дать агенту единый рабочий цикл, чтобы он за 10–15 минут входил в контекст и завершал задачу с проверяемым результатом.

## 1) Intake задачи (что читать первым)

1. `README.md` — общий контекст, scope и границы проекта.
2. `Docs/Roadmap/Agent_Roadmap.md` — что уже сделано и приоритетная очередь.
3. `Docs/Architecture/Critical_Open_Architecture_Gaps.md` — архитектурные риски и блокеры.
4. `Docs/ADR/README.md` — принятые/ожидающие архитектурные решения.
5. `Docs/Knowledge/Knowledge_Index.md` — карта остальных артефактов.

## 2) Правила исполнения

- Любое изменение должно быть атомарным: одна задача → один проверяемый результат.
- Для каждого изменения фиксировать:
  - **что сделано**;
  - **какой риск закрыт**;
  - **что остаётся открытым**.
- Если решение затрагивает архитектурные развилки, создавать/обновлять ADR.
- Не менять приоритеты roadmap молча: только с явной причиной.

## 3) Правила завершения сессии

Перед завершением задачи агент обязан:

1. Обновить `Docs/Roadmap/Agent_Roadmap.md` (статус или факт выполнения).
2. Добавить запись в `Docs/Roadmap/Agent_Handoff_Log.md`.
3. При необходимости добавить/обновить ADR в `Docs/ADR/`.
4. Указать команды проверки и фактический результат.

## 4) Единый шаблон отчёта агента

```text
Дата: YYYY-MM-DD
Задача: <коротко>
Что сделано: <1–3 пункта>
Проверка: <команды + статус>
Риски/ограничения: <если есть>
Следующий шаг: <конкретный action>
Ссылки: <файлы/коммиты/ADR>
```

## 5) Критерии готовности сессии

Сессия считается завершённой только если:

- есть обновлённый артефакт в репозитории;
- есть хотя бы одна команда проверки;
- есть запись в handoff log;
- есть понятный следующий шаг для следующего агента.
