# Индекс знаний проекта

Цель индекса — единая точка входа в ключевые документы проекта для архитектуры, контрактов и эксплуатации.

## Core

| Документ | Назначение | Когда читать | Владелец | Частота обновления |
|---|---|---|---|---|
| `README.md` | Базовый обзор проекта: цели, scope, архитектурный вектор, pipeline и принятые принципы. | При входе в проект, перед стартом любой задачи и при согласовании контекста между участниками. | Архитектор проекта / владелец репозитория. | По мере изменения стратегии и ключевых решений (обычно каждую крупную итерацию). |
| `Docs/Roadmap/Agent_Roadmap.md` | План этапов реализации и приоритетов агентной работы. | Перед планированием спринта, выбором следующей задачи и оценкой зависимостей. | Владельцы roadmap / архитектурный контур. | Не реже 1 раза в спринт и при смене приоритетов. |
| `Docs/Architecture/Critical_Open_Architecture_Gaps.md` | Реестр критических архитектурных пробелов и рисков, блокирующих развитие. | Перед внедрением изменений в архитектуру и при уточнении технического долга. | Архитектурный контур. | После каждого архитектурного review или при обнаружении нового критического пробела. |
| `Docs/ADR/README.md` и `Docs/ADR/*.md` | Журнал архитектурных решений (ADR): что и почему было выбрано, а также последствия. | Перед принятием новых архитектурных решений и при разборе причин текущих ограничений. | Архитектор(ы) решения / автор ADR. | При каждом значимом архитектурном решении. |

## Contracts

> Блок заполняется по мере появления формализованных контрактов.

| Документ | Назначение | Когда читать | Владелец | Частота обновления |
|---|---|---|---|---|
| `Docs/Contracts/Schemas/*` _(когда появятся)_ | Схемы данных (вход/выход, внутренние структуры). | Перед изменением форматов данных и интеграцией модулей. | Владельцы доменных моделей. | При каждом изменении структуры данных. |
| `Docs/Contracts/API/*` _(когда появятся)_ | Контракты API (эндпоинты, форматы запросов/ответов, версии). | Перед интеграцией клиентов и публикацией изменений API. | Владельцы API / интеграционного слоя. | При каждом изменении API, обязательно с версионированием. |
| `Docs/Contracts/Error_Catalog.md` _(когда появится)_ | Единый каталог ошибок: коды, причины, рекомендации по обработке. | При реализации/дебаге обработки ошибок и инцидентов. | Владельцы reliability/платформы. | При добавлении/изменении кодов ошибок. |
